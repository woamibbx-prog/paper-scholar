# Paper Scholar 标准化刚性指南

本文档总结了 Paper Scholar skill 的所有刚性标准化规范，以及从多次循环任务执行中积累的经验教训。

## 核心原则

**刚性标准化是确保跨周期一致性的关键**。每次执行必须严格遵守以下规范，不得有任何偏差。违反规范将导致：
- 学习循环不稳定
- 跨周期数据不一致
- 反思文件解析失败
- 知识库路径混乱
- 无法保持标准化输出

## 文件夹命名规范

### 领域文件夹
**格式**：`[领域]英文名称`

**规则**：
- 前缀 `[领域]` 后面**无空格**
- 标题仅使用英文，**禁止中文翻译**（包括括号中的中文）
- 特殊字符替换为下划线（如空格、冒号等）
- 示例：`[领域]Chain-of-Thought_Reasoning`、`[领域]Efficient_Inference`

**错误示例**：
- ❌ `[领域] Chain-of-Thought_Reasoning`（多了空格）
- ❌ `[领域]Chain-of-Thought_Reasoning_思维链`（含中文）
- ❌ `[领域]Chain-of-Thought_Reasoning (CoT)`（含英文缩写说明）
- ❌ `领域：Chain-of-Thought_Reasoning`（使用了中文冒号）

### 论文文件夹
**格式**：`[论文] 论文英文标题`

**规则**：
- 前缀 `[论文] ` 后面**必须有一个空格**
- 标题仅使用英文，**禁止中文翻译**（包括括号中的中文）
- 特殊字符替换为下划线（如空格、冒号等）
- 示例：`[论文] Attention Is All You Need`、`[论文] WizardLM_ Empowering Large Language Models`

**错误示例**：
- ❌ `[论文]Attention Is All You Need`（缺少空格）
- ❌ `[论文] Attention Is All You Need（注意机制全你需要的）`（含中文）
- ❌ `[论文] Attention_Is_All_You_Need`（全下划线）
- ❌ `论文：Attention Is All You Need`（使用了中文冒号）

### 创建后立即校验
**刚性要求**：
- 每次创建新文件夹后，**必须立即检查前缀是否正确**
- 领域文件夹 → 必须以 `[领域]` 开头，如 `[领域]Efficient_Inference`
- 论文文件夹 → 必须以 `[论文] ` 开头（注意空格），如 `[论文] Attention Is All You Need`
- 如果发现缺少前缀，**立即重命名修复**，不可留到自检阶段

## summary.md 格式规范

### 固定章节标题（一字不差）
每个 summary.md **必须使用中文**撰写，包含以下**固定章节标题**：

```markdown
# 论文中文标题

**论文ID**: arxiv:XXXX.XXXXX

## 摘要
（200字以内概述）

## 研究背景与动机
（为什么做这个研究）

## 核心方法
（关键技术和方法）

## 实验结果
（量化结果和对比）

## 局限性
（不足之处）

## 应用价值
（实际意义）

---
*由 Paper Scholar 自动总结*
```

### 禁止的标题格式
**禁止使用以下编号格式**：
- ❌ `## 一、研究背景`
- ❌ `## 二、核心方法`
- ❌ `## 3. 实验结果`
- ❌ `## 四、创新点`
- ❌ `## 五、局限性`
- ❌ `## 六、应用价值`
- ❌ `## 七、应用价值`

**必须使用**：`## 标准名` 格式

### 常见错误及修正
| 错误格式 | 正确格式 |
|---------|---------|
| `## 一、研究背景` | `## 研究背景与动机` |
| `## 二、核心方法` | `## 核心方法` |
| `## 3. 实验结果` | `## 实验结果` |
| `## 四、创新点` | `## 实验结果` |
| `## 五、局限性` | `## 局限性` |
| `## 六、应用价值` 或 `## 七、应用价值` | `## 应用价值` |

### 字数要求
- 每篇总结 800-1500 字
- 摘要 200 字以内
- 避免过短（< 800 字）或过长（> 1500 字）

## 反思文件规范

### 文件命名
**格式**：`YYYY-MM-DD_HH-MM-SS.md`（精确到秒）

**禁止格式**：
- ❌ `reflection_YYYYMMDD.md`
- ❌ `batch_xxx.md`
- ❌ `round-XX-reflection.md`
- ❌ `2026-04-17.md`（缺时分秒）

**正确示例**：
- ✅ `2026-04-17_14-30-22.md`

### 文件结构（4个PART）
必须包含以下4个PART，每个PART必须有实质内容：

#### PART 1: 本批次论文学习
对每篇论文分别列出：
- **论文标题**
  - 技术与创新点：核心技术和创新点
  - 不足：局限性或待解决的问题

**要求**：
- 每篇论文必须包含这两个子项
- 技术与创新点要具体，不能空泛
- 不足要准确，不能遗漏

#### PART 2: 结合历史反思
- **本批次在领域认知上的进展或调整**：（重要）
- 已学习领域（累计）
- 下一步领域：继续学习当前领域 还是 转向相关领域

**PART 2 深度分析要求**（关键！）：
- **禁止一句话概括**，如"本批次学习了X和Y两篇论文"
- 必须分析论文之间的技术关联、领域发展趋势、关键技术演进脉络
- 至少 3-5 句，每句都要有实质性内容
- 展示对论文的深度理解，而不是简单罗列

**错误示例**：
```
❌ 错误示例（一句话概括）
- 本批次在领域认知上的进展：本批次学习了FlashAttention和xFormers两篇论文，都是关于Transformer效率优化的。
```

**正确示例**：
```
✅ 正确示例（深度分析）
- 本批次在领域认知上的进展：本批次的两篇论文（FlashAttention和xFormers）代表了Transformer效率优化的两条关键路径——FlashAttention通过精确的IO感知注意力算法，在不牺牲精度的情况下将计算复杂度从O(n²)降低到O(n)；而xFormers则提供了一套可组合的算子库，让研究者能够灵活组合不同的优化策略。这两篇论文的技术演进脉络非常清晰：从单一算法到算子库，从理论创新到工程优化。更重要的是，它们都验证了IO优化的重要性——在GPU内存带宽受限的情况下，减少内存访问次数比减少计算量更关键。这为后续的优化研究指明了方向。
```

#### PART 3: 领域判断
- 如果新开领域：新领域与已有领域的关系（父子/兄弟/全新）
- 建议存储路径：根据领域关系，建议存放到哪个路径下，或是否需要新建文件夹

**领域关系判断**：
- 父子关系：新领域是已有领域的子领域
- 兄弟关系：新领域与已有领域属于同一父领域的并列子领域
- 全新关系：新领域与已有领域没有直接关系

#### PART 4: 下一步学习计划
**计划制定原则**：
- **具体性**：明确具体的学习方向，避免模糊表述
- **可执行性**：下一批次可以立即执行的计划
- **连贯性**：基于本批次的学习进展提出
- **格式必须性**：必须使用【标签】格式，因为 fetch 模块的 `lib.reflection_utils.parse_part4()` 通过正则匹配这些标签来提取结构化信息

**计划格式**（必须严格遵循）：
```
【继续当前领域】
- 具体方向：如 Transformer 的 FlashAttention 优化
- 学习目标：理解原理、实现、优化效果

【或转向相关领域】（如果选择继续，不要写这一段；如果选择转向，不要写上一段）
- 新领域：如 GPT 系列模型
- 学习路径：从 GPT-1 → GPT-2 → GPT-3
- 基础要求：需要先了解自回归语言模型
```

**PART 4 标签格式要求**（关键！）：
- 必须使用【】格式的标签
- 标签内容必须是：
  - `【继续当前领域】`
  - `【转向相关领域】`
  - `【或转向相关领域】`
- **禁止**其他格式，如：
  - `「继续当前领域」`（使用了中文引号）
  - `【继续】`（标签不完整）
  - `【继续本领域】`（标签名称不规范）

**正确示例**：
```
✅ 正确示例
【继续当前领域】
- 具体方向：Transformer 的 FlashAttention 优化及其实际应用
- 学习目标：理解 FlashAttention 的 IO 感知算法、实现细节、在主流模型（如 GPT、BERT）中的应用效果
```

**错误示例**：
```
❌ 错误示例（格式问题）
「继续当前领域」
- 具体方向：Transformer 的 FlashAttention 优化

❌ 错误示例（标签不标准）
【继续】
- 具体方向：Transformer 的 FlashAttention 优化
```

## 下载前去重规范

### 刚性要求
**每次下载论文 PDF 之前，必须先与 kb.json 中已有论文进行对比，禁止重复下载。**

### 执行流程
1. 调用 `lib.kb.check_duplicate_papers(kb, candidates)` 进行批量去重
2. `candidates` 为待下载论文列表，每项含 `id`（arxiv ID）和 `title`
3. 函数返回 `(new_papers, duplicates)`
4. **仅对 `new_papers` 执行下载**
5. `duplicates` 中的论文需在日志中明确列出（ID + 标题），标明「已跳过：已存在于知识库」

### 禁止操作
- ❌ **禁止**先下载再靠 `add_paper_safe` 跳过
- 必须在**网络请求前就拦截**重复论文

### 特殊处理
- 如果所有候选论文都是重复的（无新论文），跳过下载和后续步骤，直接进入反思模块告知无新论文

### 去重逻辑
- 优先按 arxiv ID 精确匹配
- 其次按标题模糊匹配（防止同一论文 ID 不同但实际是同一篇）
- 标题匹配忽略大小写和前后空格

## 周期结束前自检规范

### 刚性要求
**每个周期结束前，必须按顺序执行：**

1. **运行标准化校验**：`py -3 ~/.openclaw/workspace/scripts/paper-scholar-normalize.py`
2. **检查修复结果**：
   - 如果输出显示 `All checks passed` → 周期完成
   - 如果发现 `FIXED` 或 `BAD` 项 → **立即修复后重新运行校验，直到所有检查通过才算周期完成**
3. **输出反思文件**：将本次生成的反思文件完整输出作为回复内容，供老大审阅

### 自检内容（标准化校验脚本会检查）
- [1] 反思文件命名是否为 YYYY-MM-DD_HH-MM-SS.md
- [2] 文件夹命名是否有正确的前缀（[领域] 或 [论文]）
- [3] summary.md 格式是否符合标准（固定章节标题，无编号格式）
- [4] 是否有垃圾数据（0字节PDF、临时文件、空目录）
- [5] kb.json 路径是否与实际文件系统一致
- [6] 所有路径是否可解析

### 自检流程图
```
生成反思文件 → 运行标准化校验 → 检查结果
                                           ↓
                                   发现问题？→ 立即修复
                                           ↓
                                     重新运行校验
                                           ↓
                                   All checks passed? → Yes → 周期完成
                                                     |
                                                     No → 继续修复
```

## 论文文件夹标准内容

### 必需文件
```
[论文] 标题/
├── paper.pdf      # PDF 全文（必需）
├── summary.md     # 中文结构化总结（必需）
└── meta.json      # 元数据（必需，含 title/id/abstract/authors/published）
```

### 文件完整性要求
- **paper.pdf**：必须存在且大小 > 0 字节（0字节PDF视为无效，需重新下载）
- **summary.md**：必须存在且符合标准化格式
- **meta.json**：必须存在且包含必需字段

### 禁止文件
- extracted*.txt：临时提取文件，应删除
- 其他中间文件：应清理

## 经验教训总结

### 1. 反思格式偏离标准的问题
**问题**：在多次循环执行中，反思文件格式经常偏离标准模板，主要表现为：
- 标题错误（使用了编号格式）
- 缺少副标题（如 PART 1、PART 2 等）
- PART 结构不规范（缺少某个 PART 或内容占位符）

**教训**：
- 必须严格遵守 4 个 PART 的结构要求
- 每个 PART 必须有实质内容，不能是占位符
- PART 2 的深度分析是关键，不得一句话概括

### 2. 领域学习方向调整
**问题**：用户要求后续领域学习往 Agent 方向靠

**教训**：
- 在 PART 3 领域判断时，要考虑用户的整体学习方向
- 在 PART 4 制定学习计划时，要考虑与 Agent 相关的领域
- 可以规划学习路径：Planning → Memory → Web/Grounding → Evaluation → Safety

### 3. 论文下载不稳定问题
**问题**：多次出现下载 0 字节、SIGKILL 失败等问题

**教训**：
- 下载成功后必须检查 PDF 文件大小，0 字节视为无效
- 如果所有候选论文都是重复或下载失败，直接进入反思模块告知无新论文
- 避免浪费资源和时间

### 4. 文件夹前缀缺失问题
**问题**：创建文件夹后忘记检查前缀，导致后续批次数据不一致

**教训**：
- 每次创建新文件夹后，**必须立即检查前缀是否正确**
- 发现前缀缺失，**立即重命名修复**，不可留到自检阶段
- 周期结束前的自检是最后一道防线，不能替代创建后的立即校验

### 5. 反思文件命名不规范问题
**问题**：反思文件命名使用了 `reflection_YYYYMMDD.md` 等不规范格式

**教训**：
- 必须严格遵守 `YYYY-MM-DD_HH-MM-SS.md` 格式（精确到秒）
- 禁止使用任何其他格式
- 标准化校验脚本会自动检测并修复命名问题

## 标准化校验脚本

### 位置
`~/.openclaw/workspace/scripts/paper-scholar-normalize.py`

### 功能
1. 反思文件命名修复（YYYY-MM-DD_HH-MM-SS.md）
2. 文件夹命名修复（添加[领域]/[论文]前缀）
3. summary.md 格式统一（固定章节标题）
4. 清理垃圾数据（0字节PDF、临时文件、空目录）
5. kb.json 路径重建（与实际文件系统一致）
6. 验证所有路径可解析

### 使用场景
- **每个周期结束前必须运行**（刚性要求）
- 发现数据不一致时运行
- 数据迁移后运行

### 输出解读
- `All checks passed` → 无问题，周期完成
- `FIXED X problems` → 修复了 X 个问题
- `BAD PATH: ...` → 路径错误需手动处理

## 跨周期一致性保证

### learning_context 的作用
- 格式：3-5 句话，涵盖已学领域、关键技术脉络、当前方向
- 目的：作为跨周期的"记忆桥梁"，让新上下文能快速恢复全局认知
- 更新时机：在 complete_batch 之后、周期结束前

### 恢复协议
当在新上下文中恢复执行时（如周期性任务的第二轮），按以下步骤恢复：
1. 读取 SKILL.md：了解整体流程和模块职责
2. 检查知识库：调用 `lib.kb.load_kb()`
3. 读取最新反思：调用 `lib.reflection_utils.read_latest_reflection()`
4. 开始执行：根据解析出的计划，进入 fetch 模块

### 连续性保障
- kb.json 的 learning_context 是跨周期的"记忆桥梁"
- 只需读 3 个文件（SKILL.md + kb.json + 最新反思）即可完全恢复
- 标准化校验脚本确保数据一致性

## 总结

**刚性标准化是 Paper Scholar skill 成功的关键**。每次执行必须严格遵守以上规范，确保：
1. 文件夹命名正确
2. summary.md 格式统一
3. 反思文件结构规范
4. 下载前去重严格执行
5. 周期结束前自检通过

只有这样才能确保无论开启多少个新的循环任务，都能够保持始终如一的刚性化标准化的任务执行结果。
